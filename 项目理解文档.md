# Weblog.Core 博客项目理解文档

> 目标：帮助你从零理解这个由 AI 辅助生成的项目，为后续自己重构打好基础。

---

## 一、项目是什么

一个基于 **ASP.NET Core 8** 的博客后端 API 项目，前后端分离架构。后端只提供 RESTful API，前端（Vue/React等）单独部署。

功能包括：
- 博客文章管理（写文章、分类、标签、知识库）
- 评论系统
- 后台管理（Dashboard 统计、公告）
- AI 功能（文章摘要、AI 对话、多模型支持）
- 文件上传（MinIO 对象存储）
- 敏感词过滤

---

## 二、项目分层结构

```
Weblog.Core/src/
├── 01_Weblog.Core.Model       # 实体类 + DTO（数据定义层）
├── 02_Weblog.Core.Common      # 公共工具（JWT、统一返回结果）
├── 03_Weblog.Core.Repository  # 数据库访问层（DbContext + UnitOfWork）
├── 04_Weblog.Core.Service     # 业务逻辑层（接口 + 实现）
└── 05_Weblog.Core.Api         # Web API 入口（Controller + 中间件）
```

### 分层职责

| 层 | 职责 | 关键点 |
|----|------|--------|
| Model | 定义数据结构 | Entity（数据库表）+ DTO（接口传输对象） |
| Common | 工具类 | JWT生成/解析、`Result<T>` 统一响应格式 |
| Repository | 数据库操作 | DbContext 封装 SqlSugar 查询入口，UnitOfWork 管理事务 |
| Service | 业务逻辑 | 每个业务一个接口（IXxxService）+ 实现（XxxService） |
| Api | HTTP 入口 | Controller 接收请求 → 调 Service → 返回 Result |

## 三、核心技术栈

| 技术 | 用途 | 说明 |
|------|------|------|
| ASP.NET Core 8 | Web 框架 | 主框架 |
| SqlSugar | ORM | 类似 EF Core，但更轻量，国产 |

| Swagger | API 文档 | 开发环境自动生成接口文档页面 |

## 五、请求的完整流程

以「发布文章」为例：

```
前端 POST /api/admin/article
        ↓
AdminArticleController.CreateArticle()
        ↓  参数校验（ModelState）
        ↓  从 JWT Token 中取当前用户ID
IArticleService.CreateAsync(request)
        ↓  创建 Article 记录
        ↓  创建 ArticleContent 记录（正文单独存）
        ↓  创建 ArticleCategoryRel 记录
        ↓  创建 ArticleTag 记录（批量）
        ↓
DbContext.Db.Insertable(...).ExecuteReturnIdentityAsync()
        ↓  SqlSugar 生成 SQL，写入 MySQL
        ↓
返回 Result<ArticleDto>.Ok(dto)
        ↓
前端收到 { code: 200, data: {...} }
```

## 六、认证机制（JWT）

```
1. 登录：POST /api/login
   → AuthService.LoginAsync()
   → 验证用户名/密码（MD5）
   → JwtHelper 生成 Token（包含 userId、username）
   → 返回 Token 给前端

2. 后续请求：
   → 请求头携带 Authorization: Bearer <token>
   → ASP.NET Core 中间件自动验证
   → Controller 用 [Authorize] 标记需要登录的接口
   → 用 User.FindFirst(ClaimTypes.NameIdentifier) 取当前用户ID
```

## 七、统一响应格式

所有接口返回 `Result<T>`：

```json
// 成功
{ "code": 200, "msg": "ok", "data": { ... } }

// 失败
{ "code": 400, "msg": "用户名或密码错误", "data": null }
```

用法：`Result<ArticleDto>.Ok(dto)` 或 `Result.Fail("错误信息")`

## 八、Admin 和 Portal 的区别

Controller 分两类：

| 目录 | 路由前缀 | 说明 |
|------|----------|------|
| Controllers/Admin/ | /api/admin/... | 后台管理接口，需要 [Authorize] |
| Controllers/Portal/ | /api/portal/... | 前台展示接口，游客可访问 |

同一张表，两套 Service：
- `ArticleService` → 管理端（增删改查完整权限）
- `ArticlePortalService` → 前台端（只读，只返回已发布文章）

## 九、AI 功能模块

项目内置了一套自研的 AI 调用框架：

```
AiKernel（总入口）
  ├── AiProviderSelector   # 根据配置选择使用哪个 AI 服务商
  ├── Providers/
  │     ├── OpenAiProvider    # OpenAI / GPT
  │     ├── DeepSeekProvider  # DeepSeek
  │     ├── ClaudeProvider    # Anthropic Claude
  │     └── OtherProviders    # 其他兼容 OpenAI 协议的模型
  └── Plugins/
        └── PluginManager     # AI 插件管理（工具调用 Function Calling）
```

AI 服务商的 API Key 存在数据库（`t_ai_provider`），通过 `AiKeyEncryption` 加密存储。

## 十、重构建议（从零开始的顺序）

如果你想自己重新写一遍，建议按以下顺序：

### 第一阶段：基础搭建
1. 创建解决方案和5个项目，建好分层引用关系
2. 引入 SqlSugar，配置 MySQL 连接
3. 写 `Article` 实体，用 `CodeFirst` 自动建表
4. 写 `DbContext`
5. 写统一返回 `Result<T>`

### 第二阶段：认证
6. 写 `SysUser` 实体和登录接口
7. 配置 JWT，写 `JwtHelper`
8. 写 `AuthController`（登录 + 获取当前用户）

### 第三阶段：核心 CRUD
9. 文章的增删改查（Admin端）
10. 分类、标签
11. 文章正文分离（ArticleContent）
12. 文章与分类/标签的多对多关联

### 第四阶段：前台接口
13. 文章列表（分页、按分类/标签筛选）
14. 文章详情（阅读数 +1）
15. 评论

### 第五阶段：进阶
16. 文件上传（MinIO）
17. Dashboard 统计
18. 敏感词过滤
19. AI 摘要（可选）

## 十一、你需要重点理解的概念

| 概念 | 为什么重要 |
|------|------------|
| 依赖注入（DI） | 整个项目的核心，所有 Service 都通过构造函数注入 |
| 接口与实现分离 | `IArticleService` + `ArticleService`，便于测试和替换 |
| DTO 与 Entity 分离 | Entity 是数据库结构，DTO 是接口契约，不能混用 |
| SqlSugar 的用法 | 理解 `Queryable`、`Insertable`、`Updateable`、`Deleteable` |
| 中间件管道 | `Program.cs` 里 `app.Use*` 的顺序很重要 |
| JWT 的 Claims | Token 里存了什么、怎么取出来 |
